# Diagnosis Report

You are creating a structured diagnosis document for a system failure or data
pipeline issue. The report must be self-contained — an operator who wasn't
in the debugging session must be able to understand exactly what was wrong,
what was found in the DB, and what was done to fix it.

---

## Before You Start

Gather the following before writing:

1. **DB state at time of discovery** — actual row counts from the relevant
   tables (run the COUNT queries if you haven't already)
2. **The exact code that caused each root cause** — file path + line number
   or the SQL query, not just a description
3. **What was fixed** — the specific commits or code changes
4. **Verification commands** — the exact psql/curl/grep commands someone
   can run to confirm the fix is working

## Verification

Run after fix is deployed:

```bash
# Confirm DB rows are now being written:
psql $DATABASE_URL -c "SELECT COUNT(*) FROM game_sessions;"
# Expected: > 0 after next game sync

# Confirm JWT TTL:
grep JWT_EXPIRES_IN /opt/readyplay/api/.env
# Expected: JWT_EXPIRES_IN=7d

# Confirm public stats endpoint returns non-zero:
curl -s "https://api.readyplay.app/public/profile-stats/players/PLAYER_ID/summary" | jq .completedGames
# Expected: integer > 0 after game sync
```

## Real Example: Game Pipeline Diagnosis

The structure above is modeled on `docs/Diagnosis-Game-Pipeline.md` from the
Red-E Play project, which documented:

- DB state: `game_sessions = 0 rows` (0 of expected ~200)
- Root Cause 1: JWT TTL `1h` → silent auth failures during background sync
- Root Cause 2: `createdByPlayerID == myID` filter → non-creator participants
  never pushed their game records
- Root Cause 3: Confirmed absence of actual score-tracked games — "activity"
  was `activity_events` (lightweight signals), not `game_sessions` (records)

The report changed the debugging approach from "fix the read layer" to "fix
the push layer" — which was the only approach that could have worked.

## Common Failure Modes and Their DB Signatures

| Symptom | DB signature | Root cause category |
|---------|-------------|---------------------|
| Stats show zeros, no `game_sessions` rows | `COUNT(*) = 0` | Push layer (sync, JWT, creator filter) |
| Stats show zeros, rows exist | `COUNT(*) > 0` but wrong SQL | Read layer (participation SQL, creator-only query) |
| Recent data missing, old data present | Rows with recent timestamp = 0 | Recent push failure (JWT rotation, new device) |
| Admin shows stale data | Rows exist, API returns wrong shape | Serializer or API helper gap |
| Zeros only for some players | Rows exist for some, not others | Creator filter (only scorekeeper's data) |
| `activity_events` present but `game_sessions` = 0 | Two separate tables | Confusing feed signals for records |

## Related Rules

- `diagnosis-before-fix` — always SSH to DB first before writing code
- `stats-query-participation-not-creator` — participation SQL pattern
- `mobile-jwt-ttl` — access tokens must be ≥7d for mobile
- `all-participants-push-data` — remove creator-only push filters
- `activity-events-not-game-records` — don't confuse feed signals with records
