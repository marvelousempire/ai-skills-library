# Tool-calling approval re-entry — pause the AI loop on destructive tools

When you build an AI agent with **destructive tools** (cleanup, deletion, modification, network calls with side effects), some calls need a human gate. Auto-running everything is wrong. Forcing approval for every tool call is friction. The right design: **approval flows ONLY for tools that mutate state, and the approval card's text comes from CURATED source — never generated by the AI**.

The pattern: tools have a `requires_approval` flag. The agent loop runs normally for non-approval tools. When an approval tool is called, the SSE stream emits `tool_approval_needed`, closes cleanly, and waits. The client renders an approval card. On click, the client re-POSTs with `pending_tool_results: [{id, approved: bool}]`. The agent loop **resumes from the same message state** — same conversation history, same tool-use IDs, no replay.

## When to use this skill

- Building an AI agent with tool-calling (Anthropic Messages API or OpenAI function-calling)
- The agent has any tools that delete / modify / send / pay / install
- Auditing an existing agent that runs destructive tools without approval
- The user says "the AI just deleted my X" (give them this pattern)
- Designing the UX for BYO-API-key dev tools where the agent can act on the user's machine

## The architecture (four pieces)

1. **Tool registry with `requires_approval` flag.** Each tool dict has `tier` (A/B/C) and `requires_approval`. Tier B (destructive) is gated.

2. **Approval checker function.** `needs_approval(tool_name, args, settings) → bool`. May be settings-toggle-able (e.g. "trust the AI to run safe-tier without asking").

3. **Multi-turn loop with mid-flight pause.** The loop runs normally for Tier A. When a Tier B call is detected and `needs_approval` returns true, the loop:
   - Emits `tool_approval_needed` SSE event with the tool id, name, input, and **the curated approval-card text** pulled from source (NOT from the AI)
   - Returns `stopped_for_approval: true` with the conversation history intact
   - The caller persists the history; the client renders the approval card

4. **Re-entry endpoint.** Client re-POSTs with `messages: <same history>, pending_tool_results: [{id, approved: bool}]`. The loop:
   - Injects the approved results (or rejection messages) into the conversation
   - Resumes from the same point
   - Continues normally

## The full Anthropic-shape pattern (lifted from DustPan `web/ai.py`)

```python
def complete_with_tools(
    provider, api_key, system, messages, tools,
    tool_runner, on_event, approval_checker,
    pending_results=None, max_rounds=10,
):
    msgs = list(messages)

    # If resuming from approval, inject the pending results first
    if pending_results:
        if provider == "anthropic":
            tr_blocks = []
            for r in pending_results:
                if r.get("approved"):
                    tr_blocks.append({"type": "tool_result", "tool_use_id": r["id"],
                                      "content": json.dumps(r.get("result", {"ok": True}))})
                else:
                    tr_blocks.append({"type": "tool_result", "tool_use_id": r["id"],
                                      "content": "User rejected this action.", "is_error": True})
            msgs.append({"role": "user", "content": tr_blocks})
        # ... openai equivalent (tool message per call)

    rounds = 0
    while rounds < max_rounds:
        rounds += 1
        # ... call the model, parse tool_use blocks ...

        # ── APPROVAL GATE ────────────────────────────────────────────────────
        pending_for_approval = []
        for tu in tool_uses:
            if approval_checker(tu["name"], tu.get("input", {})):
                pending_for_approval.append({"id": tu["id"], "name": tu["name"],
                                              "input": tu.get("input", {})})

        if pending_for_approval:
            for p in pending_for_approval:
                on_event({"event": "tool_approval_needed", "data": p})
            return {
                "ok": True, "messages": msgs, "usage": usage, "rounds": rounds,
                "stopped_for_approval": True,
                "pending_calls": pending_for_approval,
            }

        # Execute non-approval tools normally
        for tu in tool_uses:
            result = tool_runner(tu["name"], tu.get("input", {}))
            on_event({"event": "tool_use_result", "data": {"id": tu["id"], **result}})
            # ... append tool_result to msgs ...

    on_event({"event": "assistant_done", "data": {"usage": usage, "rounds": rounds}})
    return {"ok": True, "messages": msgs, "usage": usage, "rounds": rounds,
            "stopped_for_approval": False}
```

## The approval card payload (curated, not AI-generated)

When `tool_approval_needed` fires, the server enriches the event with text **from the canonical source file** — not from the AI:

```python
def approval_summary(name, args, cleaners_dict):
    if name == "run_category_action":
        cid = args.get("category_id")
        aid = args.get("action_id")
        action = cleaners_dict.get(cid, {}).get("actions", {}).get(aid, {})
        return {
            "summary": f"Run '{action['label']}' in {cid}",
            "desc":    action["desc"],   # ← curated, from cleaners.py
            "cost":    action["cost"],   # ← curated, from cleaners.py
        }
```

The UI renders the card with this curated text. The user reads "One slower Xcode build. That's it." — not whatever the AI hallucinated about what the action does.

## The settings toggle (escape hatch)

For repetitive workflows, a user setting can lower the gate for safe-tier operations:

```python
def needs_approval(name, args, allow_safe_auto, cleaners_dict):
    tool = REGISTRY.get(name)
    if not tool.get("requires_approval", False):
        return False
    if not allow_safe_auto:
        return True
    # When allow_safe_auto is on: still gate non-safe tiers
    if name == "clean_path":
        path = args.get("path", "")
        cat  = cleaners_dict.get(args.get("category_id", ""), {})
        for label, p in cat.get("groups", {}).get("safe", []):
            if os.path.expanduser(p) == os.path.expanduser(path):
                return False  # safe tier + setting on → auto-approve
    return True
```

Default: setting OFF, every Tier B call requires approval. Power users can opt in.

## Hard cap on rounds

Tool-use loops can chain. Without a cap, a runaway prompt can call 1000 tools. Hardcode `max_rounds=10`. When hit, the loop emits a final `assistant_done` with a polite cutoff message:

```python
while rounds < max_rounds:
    rounds += 1
    # ... loop body ...
# Falls through naturally after max_rounds
on_event({"event": "assistant_done", "data": {"usage": usage, "rounds": rounds,
          "note": "Reached tool round cap"}})
```

## SSE event vocabulary

| Event | When | Payload |
|---|---|---|
| `provider_info` | First (per turn) | `{provider, model, tool_use_supported}` |
| `assistant_text_delta` | Each token (streaming) | `{text: <chunk>}` |
| `assistant_text` | Per-round block (non-streaming fallback) | `{text: <full block>}` |
| `tool_use_start` | Tool execution begins | `{id, name, input}` |
| `tool_use_result` | Tool execution ends | `{id, name, ok, result, elapsed_ms}` |
| `tool_approval_needed` | Tier B blocked on approval | `{id, name, input, summary, desc, cost}` |
| `assistant_done` | Loop complete | `{usage, rounds}` |
| `error` | Anything failed | `{message}` |

## Anti-patterns

- ❌ Approving everything via AI judgment alone ("if the AI says it's safe, just run it"). The approval gate is the human's job.
- ❌ AI-generated approval-card text. Hallucinations slip into safety-critical UX.
- ❌ No max_rounds cap. Runaway tool loops are real.
- ❌ Re-running the entire conversation on resume. The state is in the conversation history; re-entry just appends pending results and continues.
- ❌ Approval card without `desc` + `cost` from source. The user has to read JSON to know what's about to happen.
