# Product schema standard — the complete 8-field schema for every product

## When this fires

Any product (skill, rule, agent, template, doc, checklist) in this library or any
repo under AVERY GOODMAN is being created, modified, or reviewed. This includes
DustPan AppleScript docs (`applescripts/docs/*.md` with `DS-XXXX` IDs).

This rule encodes the decisions made in Plan 0027 (2026-05-14).

## What it enforces

### 1. Action-based keywords on every product (replaces noun/domain labels)

Keywords are `verb-object` action tags, not domain descriptions.

**Format:** `verb-object`, hyphenated, verb leads, max 2 words per tag, max 3 tags total.

```yaml
# ✅ Correct — actions the user can perform with this skill
keywords: [run-audit, check-gaps, build-leadsheet]

# ❌ Wrong — domain nouns
keywords: [audit, boolean, accountability]

# ❌ Wrong — bare verbs with no object
keywords: [audit, check, build]
```

**Rule:** The first word is always a verb. If a tag has 2 words, the second is the object.
Examples: `run-audit`, `setup-repo`, `write-skill`, `show-dialog`, `enforce-schema`.

### 2. All 8 frontmatter fields required

```yaml
---
name: product-name-kebab-case
id: SK-0042
hash: a3f9b2c
keywords: [verb-object, verb-object, verb-object]
relations: [skill-a, skill-b, skill-c]
before: [prerequisite-a, prerequisite-b]
governed_by: [RL-XXXX, global]
meta: dynamic
description: >-
  What it does. 4–6 trigger phrases embedded naturally.
---
```

### 3. The `hash` field — stable, permanent, never manually edited

`hash` is a 7-character hex string derived from `sha1(name + id)[:7]`.
Generated by `scripts/stamp-product-ids.py`. Never change it by hand.
Even if the product is renamed, the hash must stay the same — it is the
rename-proof pointer used by other products' `relations:` and `before:` fields.

### 4. The `relations` field — succession graph

List up to 3 skill names (by `name:` value, not `id:`) that logically follow this
product. These are the natural "what to do next" products. Empty `[]` is valid —
not every product has obvious successors.

```yaml
relations: [conversation-retrospective-extraction, post-ship-elevation-pass]
```

### 5. The `before` field — prerequisite graph

List up to 3 skill names that should run BEFORE this product. Empty `[]` is valid.
Do not list rules in `before:` — rules fire automatically and need no ordering.

```yaml
before: [skill-nutrients-decanter, gap-audit-and-elevation]
```

### 6. The `governed_by` field — rule linkage

List the RL-XXXX IDs of rules that specifically govern this product's behavior.
Use `global` when only library-wide defaults apply. Every product has at minimum
`governed_by: [global]`.

```yaml
governed_by: [RL-0039, RL-0044, global]   # specific rules + global defaults
governed_by: [global]                      # only library-wide defaults apply
```

### 7. The `meta: dynamic` field — living metadata signal

`meta: dynamic` tells any AI reading this product: the `keywords`, `relations`,
and `before` fields are living data — they represent best-current-knowledge and
are updated as the system learns more about how this product is actually used.

A future GitHub/GitLab Actions workflow will refresh these fields automatically
when products are updated. Until then, re-run `stamp-product-ids.py` after
significant product revisions to keep the fields current.

```yaml
meta: dynamic
```

### 8. Hybrid delivery — best available now, living over time

When adding new fields to existing products, ship the best available inferred
values immediately. Do not leave fields empty simply because the perfect value
is not known. The `meta: dynamic` flag acknowledges that values will improve.

Use `scripts/stamp-product-ids.py` to apply schema to all products at once.
The script uses lookup tables for 23+ curated skills and category-based inference
for all others.

### 9. Unified authoring standard

**SK-0021** (`create-skill`) is the single authoritative source for both schema
structure AND writing voice. Do not split authoring guidance across multiple products.

**RL-0044** (`skill-writing-voice`) exists as an enforcement rule that points to
SK-0021. It does not independently define the voice standard.

### 10. DustPan AppleScript docs (`DS-XXXX`)

Files in `applescripts/docs/*.md` in the DustPan repo follow the same schema
with `DS-XXXX` IDs. Run `stamp-product-ids.py --dustpan /path/to/dustpan` to
apply. Action keywords for AppleScript docs follow the same verb-object format:
`[run-cleanup, free-space, show-progress]`.

### 11. gap-audit-and-elevation + failure-proof-audit on every plan

Before executing any plan that modifies library products:
1. Run `gap-audit-and-elevation` (SK-0113) — identify what's missing, then elevate
2. Run `failure-proof-audit` (SK-0112) — 4-pass audit (primary → hidden → 6-month sim → future-proof)

After each ship, run both again.

## Violations

| Violation | Correct approach |
|---|---|
| `keywords: [audit, boolean, accountability]` — domain nouns | Replace with `[run-audit, check-gaps, build-leadsheet]` |
| Missing `hash:` field | Run `stamp-product-ids.py` |
| Missing `meta: dynamic` | Run `stamp-product-ids.py` |
| Empty `relations: []` when successors exist | Add the 2-3 most obvious successor skill names |
| `governed_by:` missing | Add `[global]` at minimum |
| `before:` missing | Add `[]` at minimum (empty is valid) |
| Skill voice uses "might", "consider", "generally" | Rewrite as direct commands per SK-0021 |

## Why

Codified from Plan 0027 (2026-05-14). The noun-based keyword system made the
router match by domain (what world a skill lives in) instead of by capability
(what the user can DO with it). The new action-based system enables matching by
intent and action rather than category.

The 5 new fields (`hash`, `relations`, `before`, `governed_by`, `meta`) build the
relationship graph that lets the distribution model sequence products intelligently
— routing from prerequisites through the main skill and into natural successors.
